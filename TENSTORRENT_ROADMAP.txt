BarraCUDA Tenstorrent Backend — Roadmap
=======================================

Status: AMD GFX11 backend COMPLETE (15,117 lines, zero LLVM, zero decode failures).
        Tenstorrent branch exists but empty: https://github.com/Zaneham/BarraCUDA/tree/Tenstorrent
        ISA documentation, SFPI source, and encoding tables all collected locally.

Target: Wormhole B0 initially. Blackhole later (encoding differences are minor).

Architecture summary: Tenstorrent is MIMD, not SIMT. Each Tensix tile has 5 Baby
RISC-V cores (reader, writer, unpack, math, pack) + an SFPU vector unit (32 lanes,
FP32/INT32) + an FPU matrix unit (32x32 tiles). No caches. No warp scheduler. No
automatic instruction fetch. The RISC-V cores push 32-bit instructions to the SFPU
via memory-mapped stores. Data moves through L1 SRAM (1.5 MB per tile) via explicit
NoC DMA. Kernels are split into three: reader (DRAM->L1), compute (L1->registers->L1),
writer (L1->DRAM), connected by circular buffers with semaphore sync.

This is a fundamentally different beast from AMD. The IR is target-independent but
the backend needs to solve problems that don't exist on GPUs: kernel fission, thread
coarsening (millions of CUDA threads -> tens of Tensix cores), tile-granularity data
movement, and predicated execution instead of branches.


TIER 0 — Scaffolding
---------------------
Get the new files in place, decide the output format, wire up the CLI.

0a. OUTPUT FORMAT DECISION
    Two options:
      A) Generate TT-Metalium C++ (3 kernel files + host program). Leverages existing
         runtime for reader/writer/dispatch. We just need to generate the compute kernel's
         SFPU instruction sequence as inline asm or .ttinsn directives.
      B) Generate raw RISC-V + SFPU binary. True compiler output. Much harder: need
         ELF targeting rv32im, instruction buffer setup, NoC register programming.
    Start with (A). It's honest — we generate real machine instructions for the compute
    path, and use the runtime for data movement. Graduate to (B) when we have hardware
    to test on.

0b. NEW FILES (src/tensix.h, src/tensix_isel.c, src/tensix_emit.c)
    Mirror the AMD backend structure:
      tensix.h       — machine instruction types, SFPU opcode enum, register model
      tensix_isel.c  — BIR -> SFPU instruction selection
      tensix_emit.c  — register allocation + 32-bit encoding + output emission
    Estimated: ~200 lines for the header, stubs for the rest.

0c. CLI FLAG (--tensix or --tenstorrent-bin)
    Add to main.c alongside --amdgpu-bin. Calls tensix_compile() instead of
    amdgpu_compile(). Wire through the existing pipeline: lex -> preproc -> parse ->
    sema -> bir_lower -> mem2reg -> tensix_isel -> tensix_emit.
    ~30 lines in main.c.

0d. OPCODE TABLE (tensix.h)
    Enumerate the SFPU opcodes we actually need. Not all 130+. Start with:
      Arithmetic:  SFPADD(0x85), SFPMUL(0x86), SFPMAD(0x84), SFPIADD(0x79)
      Immediate:   SFPADDI(0x75), SFPMULI(0x74), SFPLOADI(0x71)
      Bitwise:     SFPAND(0x7E), SFPOR(0x7F), SFPXOR(0x8D), SFPNOT(0x80), SFPSHFT(0x7A)
      Comparison:  SFPSETCC(0x7B)
      Data:        SFPLOAD(0x70), SFPSTORE(0x72), SFPMOV(0x7C)
      Control:     SFPENCC(0x8A), SFPPUSHC(0x87), SFPPOPC(0x88), SFPCOMPC(0x8B)
      Convert:     SFPCAST(0x90), SFPEXEXP(0x77), SFPEXMAN(0x78)
      Misc:        SFPABS(0x7D), SFPSETSGN(0x89), SFPDIVP2(0x76), SFPLZ(0x81)
    ~40 opcodes to start. Add more as needed.

0e. REGISTER MODEL (tensix.h)
    8 general-purpose vector registers: L0-L7 (32 lanes x 32 bits each).
    Constant registers: L8=0.8373, L9=0, L10=1.0, L11-L14=programmable, L15=laneID.
    Dst register file: 1024x16 rows (spill target, also used for Dst<->SFPU data movement).
    GPRs: 64 per RISC-V thread (for scalar address computation, loop counters).
    With only 8 GPRs we'll spill aggressively to Dst. That's fine — 1 cycle SFPLOAD/SFPSTORE.


TIER 1 — SFPU Instruction Selection
------------------------------------
Make the compute path work. Take BIR, produce SFPU machine instructions. This is the
direct analogue of amdgpu_isel.c but for a 32-lane vector unit instead of a SIMT wavefront.

1a. FLOATING-POINT ARITHMETIC (tensix_isel.c, ~300 lines)
    BIR_FADD -> SFPADD (opcode 0x85, Format A: [src_a:4][src_b:4][src_c:4][_:4][dst:4][mod1:4])
    BIR_FSUB -> SFPADD with mod1 flag for negation (or SFPIADD on the sign bit? TBD)
    BIR_FMUL -> SFPMUL (opcode 0x86, same format)
    BIR_FDIV -> SFPMUL + reciprocal approximation (SFPLUT or SFPARECIP on Blackhole)
    BIR_FREM -> software sequence (not hardware supported)
    Immediate forms: SFPADDI (0x75, Format C: [imm16:16][dst:4][mod1:4]) for add-by-constant.
    FMA: detect add(mul(a,b),c) pattern -> SFPMAD (0x84, Format A: dst = src_a * src_b + src_c).

1b. INTEGER ARITHMETIC (tensix_isel.c, ~200 lines)
    BIR_ADD  -> SFPIADD (opcode 0x79, Format B: [imm12:12][src_c:4][dst:4][mod1:4])
    BIR_SUB  -> SFPIADD with negated operand or subtract mod
    BIR_MUL  -> SFPMUL24 (opcode 0x98, Blackhole only) or software mul via shifts+adds
    BIR_AND  -> SFPAND (0x7E), BIR_OR -> SFPOR (0x7F), BIR_XOR -> SFPXOR (0x8D)
    BIR_NOT  -> SFPNOT (0x80)
    BIR_SHL/LSHR/ASHR -> SFPSHFT (0x7A), shift direction via mod1
    Problem: SFPIADD has a 12-bit immediate, not a 3-operand form. For reg+reg add,
    need to use the dst register trick: SFPIADD with imm=0 adds src_c to dst.
    This is the trickiest mapping — SFPU integer ops are limited compared to AMD VOP3.

1c. COMPARISON + PREDICATION (tensix_isel.c, ~200 lines)
    No branch instructions in SFPU. All control flow is predicated.
    BIR_FCMP -> SFPSETCC (0x7B): sets per-lane condition code flags
    BIR_ICMP -> SFPSETCC with integer comparison mode
    BIR_BR_COND -> SFPENCC (0x8A) to enable conditional execution
    BIR_BR (after if/else) -> SFPPOPC to restore mask
    If/else pattern:
      SFPSETCC(cond)       -- set flags
      SFPENCC(mode)        -- enable predication
      [then-body]          -- only active lanes execute
      SFPCOMPC             -- complement mask (flip to else-lanes)
      [else-body]
      SFPPOPC              -- restore previous mask
    Nested ifs: SFPPUSHC/SFPPOPC stack. Hardware supports it.
    This replaces AMDGPU's exec mask + s_cbranch. Simpler in some ways, weirder in others.

1d. DATA MOVEMENT (tensix_isel.c, ~150 lines)
    BIR_LOAD/STORE (global address space) -> will be handled by reader/writer kernels (Tier 2).
    BIR_LOAD/STORE (Dst register file) -> SFPLOAD(0x70) / SFPSTORE(0x72)
      Format D: [lreg:4][mod0:4][addr_mode:2][dest_reg_addr:14]
      mod0 selects format: FP32(3), FP16A(1), FP16B(2), SrcB(0)
    BIR_ALLOCA -> map to Dst row allocation (track cumulative row offset per function)
    Constants: SFPLOADI(0x71) loads 16-bit immediate into LReg.
      For 32-bit constants: SFPLOADI upper half + SFPLOADI lower half (two instructions).

1e. CONVERSION (tensix_isel.c, ~100 lines)
    BIR_FPTOSI/FPTOUI -> SFPCAST (0x90) with appropriate mod1
    BIR_SITOFP/UITOFP -> SFPCAST reverse direction
    BIR_FPTRUNC -> SFPCAST (FP32 -> BF16/FP16)
    BIR_FPEXT  -> SFPCAST (BF16/FP16 -> FP32)
    BIR_TRUNC/ZEXT/SEXT -> SFPAND mask or SFPSHFT sign-extend sequence


TIER 2 — Register Allocation + Encoding
----------------------------------------
Take the selected instructions, assign registers, encode to 32-bit words.

2a. REGISTER ALLOCATOR (tensix_emit.c, ~600 lines)
    Only 8 vector registers (L0-L7). This is brutally few compared to AMD's 256 VGPRs.
    Spill target: Dst register file (1024 rows x 16 cols). 1-cycle SFPLOAD/SFPSTORE.

    Strategy: Fortran H two-pass activity-based allocation (adapted from IBM's 1968
    register allocator for System/360, which also had ~8 usable registers).

    Reference: C:\dev\vintage\vintage-compilers\fortran-h\source\
      iekrfp.mac (FWDPAS — forward pass, table building)
      iekrf1.mac (FWDPS1 — operand analysis, activity scoring)
      iekrbp.mac (BKPAS — backward pass, register assignment)
      iekrfr.mac (FREE — register eviction/downgrade)
      iekrgb.mac (GLOBAS — loop-level global assignment)

    The Fortran H algorithm in three passes, adapted for SFPU:

    PASS 1 — FORWARD SCAN (build tables)
      Walk BIR instructions forward through each basic block.
      For each SSA value, compute:
        - activity: weighted score based on usage (store=1, fetch=2, binary_op=3, fma_op=4)
        - def_point: instruction index where value is defined
        - last_use: instruction index of last use in block
        - busy_on_exit: does this value need to survive the block boundary?
      Build candidate table: only values that are defined AND used within the same block
      are candidates for LReg assignment. Values only used once, or only loaded for
      a single instruction, get lower priority.
      Track store/fetch pattern per variable (Fortran H's MVF/MVS bitfields).

    PASS 2 — BACKWARD SCAN (assign registers)
      Walk instructions backward using the tables from Pass 1.
      For each operand:
        a) Already assigned an LReg? Reuse it.
        b) Free LReg available? Assign it.
        c) No free LReg? EVICT the lowest-activity current assignment.
           Eviction = insert SFPSTORE to spill the evicted value to a Dst row,
           then SFPLOAD to reload it when needed later.

      BUSY-ON-EXIT THRESHOLD (the key Fortran H insight):
        If a value must be in Dst at the end of the block anyway (e.g. it's live-out
        to another block, or it feeds a pack operation), only assign an LReg if the
        local activity score >= threshold (Fortran H uses 7). Otherwise, just leave it
        in Dst and load it for each use. This avoids wasting a register on something
        that needs a store regardless.
        With 8 LRegs and 1-cycle spill, the threshold should be ~4 uses.

      OPERAND SHARING (another Fortran H trick):
        If operand 1 of an instruction (the result) has no register assigned yet,
        and operand 2 or 3 already has one, reuse that register for the result.
        This avoids a move. Works for SFPADD/SFPMUL where one source can be
        overwritten. SFPMAD(dst = a*b+c) can reuse c's register for dst.

    PASS 0 (OPTIONAL) — LOOP-LEVEL GLOBAL ASSIGNMENT
      Fortran H's GLOBAS. For tile loops where a value is live across all iterations
      (e.g. an accumulator), pin it to an LReg for the entire loop.
      Sort by activity, assign highest-activity values first.
      Remaining registers go to the per-block local allocator.
      This is the same as "global register allocation" but at tile-loop granularity.
      Defer to Tier 6 if the simpler per-block allocator works well enough initially.

    Data structures:
      - lreg_state[8]: what SSA value is in each LReg, or 0 if free
      - dst_slots[]: Dst row allocation for spills (cumulative, like AMD's scratch)
      - activity[]: per-value activity score from forward pass
      - busy_exit[]: bitfield, which values are live-out from block
    ~600 lines estimated, up from 400, because the two-pass approach is worth the extra
    complexity. With only 8 registers, a naive linear scan wastes too many on low-value
    assignments. The 1968 IBM engineers figured this out. We should listen.

2b. BINARY ENCODING (tensix_emit.c, ~400 lines)
    Simple compared to AMD. All instructions are exactly 32 bits.
    Encoding: (opcode << 24) | params
    Format A (3-src):  [op:8][src_a:4][src_b:4][src_c:4][_:4][dst:4][mod1:4]
    Format B (imm12):  [op:8][imm12:12][src_c:4][dst:4][mod1:4]
    Format C (imm16):  [op:8][imm16:16][dst:4][mod1:4]
    Format D (ld/st):  [op:8][lreg:4][mod0:4][addr:2][dest:14]
    Format E (loadi):  [op:8][lreg:4][mod0:4][imm16:16]
    No variable-length instructions. No literal constants. No VOPD dual-issue.
    The AMD encoder was 1,735 lines of suffering. This should be ~400 lines of mild discomfort.

2c. OUTPUT EMISSION (tensix_emit.c, ~300 lines)
    For Tier 0's option (A): generate TT-Metalium C++ compute kernel.
    Output format:
      #include "compute_kernel_api.h"
      namespace NAMESPACE {
      void MAIN {
          // unpack/pack boilerplate
          for (int tile = 0; tile < num_tiles; tile++) {
              cb_wait_front(cb_in, 1);
              tile_regs_acquire();
              // BEGIN generated SFPU instruction sequence
              TTI_SFPLOADI(L0, SFPLOADI_MOD0_FLOATA, 0x3F80);  // 1.0f
              TTI_SFPADD(L0, L1, L2, L3, 0);
              // ... etc
              // END generated sequence
              tile_regs_commit();
              tile_regs_wait();
              pack_tile(0, cb_out);
              tile_regs_release();
              cb_pop_front(cb_in, 1);
              cb_push_back(cb_out, 1);
          }
      }
      }
    The TTI_* macros expand to `.ttinsn` pushes. This is real code that runs on hardware.


TIER 3 — Thread Model Mapping
------------------------------
The fundamental translation: CUDA's "millions of scalar threads" becomes "tens of cores,
each processing tiles of 32 elements via a 32-lane vector unit."

3a. THREAD ID LOWERING (tensix_isel.c, ~100 lines)
    BIR_THREAD_ID(x) -> SFPU lane index (L15 gives lane IDs: 0,2,4,...,62; divide by 2)
    BIR_THREAD_ID(y) -> 0 (single SFPU lane is 1D, no y dimension)
    BIR_BLOCK_ID  -> core coordinate (passed as kernel argument by host program)
    BIR_BLOCK_DIM -> 32 (hardcoded: one SFPU width)
    BIR_GRID_DIM  -> total core count (passed as kernel argument)
    For kernels that use threadIdx.x + blockIdx.x * blockDim.x:
      lane_id + core_id * 32 gives the global element index.

3b. THREAD COARSENING ANALYSIS (new: tensix_coarsen.c, ~500 lines)
    CUDA: 1M threads, each does c[idx] = a[idx] + b[idx]
    Tensix: 64 cores, each processes 16K elements = 16 tiles of 1024 elements
    Transformation:
      1. Detect the "one thread per element" pattern (threadIdx + blockIdx * blockDim)
      2. Replace with tile loop: for (tile = 0; tile < tiles_per_core; tile++)
      3. Within each tile iteration, the SFPU's 32 lanes process 32 elements
      4. For 1024-element tiles (32x32), need 32 iterations of the 32-lane SFPU
    Alternatively, for 1D operations, each SFPU pass handles 32 elements and we loop.
    The coarsening factor = (total_elements / num_cores / 32).
    This is the hardest part of the whole backend. It changes the structure of the BIR.

3c. GRID DIMENSION MAPPING (tensix_coarsen.c, ~100 lines)
    CUDA grid: gridDim.x * blockDim.x threads
    Tensix grid: NxM core grid (e.g. 8x8 = 64 cores on Wormhole)
    Mapping: flatten CUDA grid to 1D, divide by (32 * tiles_per_core) to get core count.
    Host program configures the core range and passes per-core tile count as argument.
    Generate the host-side SetRuntimeArgs() calls.


TIER 4 — Data Movement
-----------------------
Generate the reader and writer kernels. This is where Tenstorrent diverges most
from GPU compilers. On a GPU, load/store just works. Here, data movement is a
separate program running on a separate processor.

4a. READER KERNEL GENERATION (new: tensix_datamov.c, ~400 lines)
    Analyse the BIR for global memory loads. For each:
      - Determine DRAM source address (from kernel argument)
      - Calculate tile count and stride
      - Generate NoC read sequence: noc_async_read() per tile
      - Wrap in circular buffer protocol: cb_reserve_back / cb_push_back
    Output: a C++ reader kernel file.
    For simple kernels (vector_add): template with address calculation from args.
    For complex kernels: need to analyze access patterns.

4b. WRITER KERNEL GENERATION (tensix_datamov.c, ~300 lines)
    Mirror of reader: global stores become NoC writes.
    cb_wait_front / noc_async_write / cb_pop_front.
    One writer per output buffer.

4c. HOST PROGRAM GENERATION (tensix_datamov.c, ~300 lines)
    Generate the TT-Metalium host program:
      - Device init
      - Buffer allocation (DRAM interleaved)
      - Circular buffer creation (one per kernel input/output)
      - Program creation with reader/compute/writer kernels
      - Runtime args: DRAM addresses, tile counts, core coordinates
      - Launch and readback
    This is boilerplate-heavy but straightforward template generation.

4d. DATA LAYOUT (tensix_datamov.c, ~200 lines)
    CUDA assumes row-major. Tenstorrent wants 32x32 tiles in face-transposed order.
    For element-wise ops: can treat data as flat 1D, load 32 elements at a time.
      No layout conversion needed. Just read 32 floats per SFPU pass.
    For 2D operations (matmul, stencil): need tile layout conversion.
      Insert host-side tilize/untilize calls.
      Or do it in the reader/writer kernels (more flexible).
    Start with 1D-flat and add tiling when we need matmul.


TIER 5 — Advanced Patterns
---------------------------
Handle CUDA features that don't map cleanly to the Tensix model.

5a. BARRIERS + SYNCHRONIZATION (~100 lines in tensix_isel.c)
    BIR_BARRIER -> within a single core, this maps to cb_push_back / cb_wait_front
    between the unpack/math/pack phases. For cross-core sync, need NoC-based semaphores.
    Start simple: __syncthreads() -> no-op if entire block maps to one core.
    If block spans cores: error for now, implement NoC sync later.

5b. REDUCTIONS (~200 lines)
    BIR_SHFL_DOWN -> no hardware shuffle. Tenstorrent SFPU lanes can't talk to each other.
    Workaround: SFPSTORE to Dst, then SFPLOAD with different row addressing.
    Pattern: store all lanes to Dst rows, then load with stride to accumulate.
    This is slow (many cycles) but correct. A 32-element reduction needs ~5 rounds.
    Cross-core reduction: generate a second program phase where one core gathers via NoC.

5c. ATOMICS (~150 lines)
    BIR_ATOMIC_ADD -> no hardware atomics on the compute path.
    Strategy 1: "owner-computes" — each core owns a region, no contention.
    Strategy 2: L1 atomics via Scalar Unit (ATINCGET, ATCAS) for intra-core.
    Strategy 3: serialize on a designated core via NoC for global atomics.
    For histograms: privatize per-core, merge at end. Correctness first, speed later.

5d. SHARED MEMORY (~100 lines)
    BIR_SHARED_ALLOC -> allocate from L1 SRAM (1.5 MB per core, vastly more than GPU's 48 KB).
    If the entire CUDA block maps to one core: shared memory lives in L1, trivially.
    Access: through the Scalar Unit (GPR-addressed L1 loads/stores) or unpack into Dst.
    __syncthreads after shared memory writes -> just a STALLWAIT to drain the pipe.

5e. WARP VOTES (~50 lines)
    BIR_BALLOT -> read the SFPU condition code register as a 32-bit mask.
    BIR_VOTE_ANY -> OR-reduce of condition codes (SFPU can do this).
    BIR_VOTE_ALL -> AND-reduce of condition codes.
    These actually map reasonably well since SFPU has per-lane condition codes.


TIER 6 — Optimisation
----------------------
Make the generated code not embarrassing.

6a. INSTRUCTION SCHEDULING (~300 lines)
    SFPU FP ops are 2-cycle latency, 1 IPC. Integer/bitwise are 1-cycle.
    Interleave independent FP ops to hide the 2-cycle latency.
    Dst register has a 4-cycle RAW hazard — schedule 4+ instructions between
    SFPSTORE and SFPLOAD to the same Dst row.

6b. REGISTER PRESSURE REDUCTION (~200 lines)
    With only 8 LRegs, register pressure is the bottleneck.
    Rematerialization: re-compute cheap values instead of spilling.
    Constant hoisting: use L9(0), L10(1.0), and program L11-L14 for kernel constants.
    Live range splitting: break long live ranges at natural boundaries.

6c. TILE LOOP PIPELINING (~200 lines)
    Overlap unpack of tile N+1 with compute of tile N with pack of tile N-1.
    The three RISC-V cores (T0/T1/T2) naturally support this.
    Need to generate the circular buffer depths and semaphore counts correctly.

6d. MOP/REPLAY EXPANSION (~100 lines)
    SFPMOP and SFPREPLAY are hardware loop mechanisms.
    MOP: one instruction expands to a sequence of backend ops.
    REPLAY: hardware unrolls a small instruction loop.
    Use these to reduce instruction push overhead for repetitive patterns (tile loops).

6e. MULTI-CORE DISTRIBUTION (~200 lines)
    Currently: one kernel = one core. Need to distribute across the grid.
    Generate host program that assigns tile ranges to cores.
    Load balancing: divide tiles evenly, handle remainder on last core.
    For non-uniform workloads: work-stealing via L1 atomic counters.


TIER 7 — Matrix Unit (FPU) Integration
---------------------------------------
The big prize. SFPU is a vector engine (32-lane, scalar-ish ops).
The FPU is a matrix engine (32x32 tile matmul in 5 cycles). For ML workloads,
the FPU is where the TFLOPS live.

7a. MATMUL DETECTION (~300 lines)
    Detect tiled matmul pattern in BIR: nested loops with multiply-accumulate.
    Classic pattern: for i, for j, for k: C[i][j] += A[i][k] * B[k][j]
    This is a well-studied compiler problem (polyhedral analysis, pattern matching).
    Start with explicit pattern matching on the BIR loop structure.

7b. FPU INSTRUCTION GENERATION (~400 lines)
    MVMUL: Dst[8x16] += SrcB[8x16] @ SrcA[16x16]. The fundamental op.
    Needs: unpacker setup (SrcA/SrcB from L1), MVMUL instruction, packer to write Dst back.
    Fidelity phases: 1-4 passes depending on precision needs (BF16 vs FP32).
    Generate the full unpack -> compute -> pack sequence with proper semaphores.

7c. TILING + DATA LAYOUT (~300 lines)
    Convert CUDA's row-major matrices to Tenstorrent's 32x32 tile layout.
    Face order: each 32x32 tile = four 16x16 faces in a specific memory layout.
    Generate the tilize/untilize transformations in the reader/writer kernels.
    Handle non-32-aligned matrix dimensions with padding.


SUGGESTED ORDER
---------------
Each tier is roughly one session-sized chunk:

  0a -> 0b -> 0c -> 0d -> 0e       (scaffolding: one session)
  1a -> 1b -> 1c -> 1d -> 1e       (isel: one or two sessions)
  2a -> 2b -> 2c                   (emit: one session)
  3a -> 3b -> 3c                   (thread model: one session)
  4a -> 4b -> 4c -> 4d             (data movement: one session)
  5a -> 5b -> 5c -> 5d -> 5e       (advanced: one or two sessions)
  6a -> 6b -> 6c -> 6d -> 6e       (optimisation: ongoing)
  7a -> 7b -> 7c                   (matrix unit: one or two sessions)

Tiers 0-2 get us to "it compiles element-wise CUDA to real SFPU instructions."
Tier 3 gets us to "the thread model makes sense on Tensix."
Tier 4 gets us to "the data actually moves."
Tiers 5-7 get us to "it handles real workloads."

Realistically, Tiers 0-4 should be achievable before needing hardware to test.
After that, post it on the Tenstorrent Discord and see if someone will let us
run it on actual silicon. The encoding has been documented to death by corsix.org
and the SFPI source, so there's no excuse for getting the bits wrong.
